The new ID scheme

Two flat global counters — no categories, no padding, never reused

Old (in context/plans/)
§1.1_thinking-learning-upgrade.exp.md
EXP-001-architecture.md

Special chars in URLs, ID changes when phase advances, status baked into the file name.

New (in context/lab/)
P-1 /plans/p-1-lab-bootstrap/exploration
P-32 /plans/p-32-something/context
D-3 /decisions/d-3-async-cli

Stable ID, phase as page suffix, status as frontmatter property.

Prefix Used for Examples
P-<N> Plans P-1, P-2, P-32
D-<N> Decisions / ADRs D-1, D-3, D-123

No categories. No zero-padding. No reuse. Topic, area, and team belong in tags or in the slug, not in the ID.

Status is frontmatter, not a folder

A plan never moves between folders — only its status field changes

Don't
plans/
├── active/
│   └── p-1-lab-bootstrap/
└── finished/
    └── p-2-something/

Renaming a folder breaks every URL, every cross-link, and every bookmark.

Do
plans/
├── index.mdx          # filtered live
├── p-1-lab-bootstrap/ # status: active
└── p-2-something/     # status: finished

Status changes are a one-line frontmatter edit. Path stays stable forever.

The /plans/ index page reads status from every plan's frontmatter at build time and renders one filtered table per status group:

---
plan_id: P-1
status: active        # ← the only thing that changes when a plan finishes
phase: ctx
created: 2026-05-02
---

Implemented as a small React component in context/lab/src/components/PlanList.tsx that consumes usePluginData('docusaurus-plugin-content-docs').

Folder per plan, files per phase

Each plan gets its own Docusaurus sidebar group

context/lab/docs/plans/p-1-lab-bootstrap/
├── _category_.json    # label: "P-1 Lab Bootstrap"
├── context.md         # sidebar_position: 1
├── exploration.md     # sidebar_position: 2
├── insights.md        # sidebar_position: 3
└── production.md      # sidebar_position: 4 (created after G2)

Each phase file carries this frontmatter: 

---
id: p-1-exploration
title: Exploration
sidebar_position: 2
plan_id: P-1
plan_slug: lab-bootstrap
phase: exp           # ctx | exp | prd | insights
status: active       # active | finished | archived | deprecated
created: 2026-05-02
---

Workflow

Build the Lab locally, commit the build output, push to ship

1

Author or update a plan

Create a plan folder with the new ID scheme. Each phase is its own markdown file.

$ mkdir -p context/lab/docs/plans/p-2-some-topic
$ $EDITOR context/lab/docs/plans/p-2-some-topic/context.md
2

Preview locally

Live-reload at localhost:3000. Edits show immediately.

$ cd context/lab
$ npm run start
3

Build into docs/flow/

Static output goes straight into the gh-pages root.

$ cd context/lab
$ npm run build      # writes to ../../docs/flow/
4

Commit & push

Both source and build are versioned together — reviewers can diff content and rendered output.

$ git add context/lab/ docs/flow/
$ git commit -m "lab: add p-2 context"
$ git push            # gh-pages serves automatically

Strategy: the lab-curator agent

A specialised subagent handles ID assignment, scaffolding, and build — you just describe the plan

Defined in agents/lab-curator.md. You invoke it whenever you want to:

  • • Create a new plan or decision — the agent picks the next free P-<N> or D-<N>, scaffolds the folder, fills frontmatter.
  • • Promote a plan from one phase to the next — e.g. exploration.md exists, agent creates production.md with linked frontmatter.
  • • Change a plan's status — one frontmatter edit across all phase files; folder stays put.
  • • Build & preview — agent runs npm run build, reports broken links, missing sidebar entries.

The agent never touches the rest of context/ or docs/*.html. Its blast radius is context/lab/ and docs/flow/ only.

Kill criteria

When to abandon the experiment

The Lab is killed and the context/lab/ + docs/flow/ folders are deleted if any of the following holds after 4 weeks:

  • • Authoring a plan in the Lab takes more time than the markdown-only flow in the rest of context/.
  • • The build output bloats the repo by > 50 MB.
  • • Search and navigation in the Lab are not measurably better than grep in context/plans/.
  • • The frontmatter status filter proves harder to use than just naming subfolders active/ / finished/ after all.

If the experiment succeeds, a follow-up plan migrates the existing context/plans/ into the Lab schema and retires the old §-numbering.
