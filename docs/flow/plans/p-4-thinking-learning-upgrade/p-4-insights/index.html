Insights — P-4 Framework Thinking & Learning Upgrade

Hypotheses tested​

#Hypothesis (from the plan)MethodResultVerdict
1"The framework can be extended with an uncertainty/learning layer without breaking the existing 3-step lifecycle."Self-hosting: implement Steps 1–11 inside chevp-ai-framework itself, observe whether existing artifacts still work.All existing files still parse and link; no template was deleted; the existing gate-validator was kept as a dispatcher; existing CTX/EXP/PRD plans still satisfy the new minimum-substance rule (after adding evidence: and Kill Criteria).confirmed
2"Out-of-scope items can be retained as Plan Proposals at near-zero cost, instead of being lost or causing scope bloat."Designed gatekeeper-g* outputs to spawn proposals automatically; created folder structure; defined max-5 cap and 90-day auto-defer to bound spam and backlog.Mechanism in place but untested at runtime — no actual gate-check has been run yet on a real plan that produced proposals.inconclusive
3"An internal Challenger role can produce specific, non-stereotyped critique on the first plan it sees."Wrote a Challenger section for P-4 itself (this plan) and judged whether the output is concrete or generic.The P-4 Challenger output names ceremony inflation (with a 3× metric), Challenger decay (with a similarity metric), and proposal graveyards (with a 30-day metric). All three are specific to this plan, not template-fillable. The strongest counter-argument concedes a real risk (problem-not-felt-by-users) instead of strawmanning.confirmed for n=1 — but the role's durability across the next 10 plans is what actually matters and is still untested.

What surprised us​

  • The §-numbering system created naming friction that the plan did not anticipate. When step IDs inside the plan were also written as §1–§11, they collided visually with the §-tree's plan IDs (§1.1, §1.2). Renaming to Step 1–Step 11 was a free fix in retrospect, but it surfaced only when the user pointed out the §-system was being ignored — it would have been a slow source of confusion otherwise.
  • The provenance rule (AI cannot write decided-by/approved-by) created an interesting tension with the user's instruction to "umsetze" the plan. The implementation proceeded without approval recorded, on the basis that verbal user instruction is the approval action — but the formal /approve step is still owed. This is the framework working as designed, even if it feels procedural.
  • Writing the Challenger output for P-4 was the most useful exercise of the whole plan. The "strongest counter-argument" forced a concession that the motivation came from one critique, not observed user pain. That admission is now load-bearing for P-5's design — if P-5 cannot find a real user who will fill these fields, P-4 should be reconsidered.

What we now believe​

  • The Thinking & Learning layer can coexist with the execution lifecycle. The risk is not technical compatibility but adoption friction.
  • The Gatekeeper / Plan Proposal mechanism is the highest-leverage idea in P-4 because it converts "out of scope" from a loss into a queue. But the value is conditional on humans actually triaging the queue — which the auto-defer rule alone may not be enough to ensure.
  • The Challenger role's durability is the single biggest open question. It works for n=1; it must be re-evaluated by reading 3–5 future Challenger outputs back-to-back.

What we still do not know​

  • Whether average time-to-G2 actually grows >3× on consumer repos (the kill criterion fires here)
  • Whether human reviewers will actually /promote and /reject proposals, or let the folder rot
  • Whether the new evidence: block produces meaningful entries or boilerplate
  • Whether exploration-mode: A is ever used in practice, or whether teams skip straight to B and treat A as overhead

These are all open hypotheses for P-5 — Consumer Rollout (the next plan in this thread).

Consequence for the plan​

  • Plan unchanged — proceed to Production (P-5 is the rollout plan; P-4 itself ships as a self-contained framework upgrade in this commit)
  • Plan adjusted
  • Plan killed
  • Falling back to Context

Notes on the self-hosting pilot​

This insights file is itself the first artefact produced under the new rules. If you can read it without confusion, the new template is workable. If anything in this file feels like ceremony, that is the signal P-4 should worry about — please record that observation in risks.md so it can feed P-5's risk table.
