# Challenger Output — EXP-001 chevp-flow Architecture

> Per Rule #11: the Challenger role produces an honest critique of the proposed Exploration plan before G2. No politeness, no strawmen, no rubber-stamping. The framework's value depends on this being adversarial.

## Top-3 Failure Modes (concrete, not generic)

### FM-1: `claude -p` JSON-stream schema regression silently produces empty `result` events

**Concrete scenario.** Anthropic ships a Claude Code update that renames or restructures stream-json events — e.g., adds a new required wrapper around `result.text`, or splits `text` events into `message_start`/`content_delta`/`message_stop`. Our `internal/runtime/messages.go` parses successfully (Go's `encoding/json` silently ignores unknown fields by default). Tests pass against captured fixtures. Production runs of `chevp gate-check G1` produce verdicts where the actual `VERDICT:` line is empty — but the parser sees an empty string and returns `pass` because no `block` keyword is present.

**Why this is a real risk.** `claude -p` output is not formally versioned. Anthropic ships ergonomic improvements without semver discipline. Go's permissive default-decode behaviour is a documented gotcha. The combination is silent governance failure.

**Why this matters.** Users approve gates that should have blocked. The framework's enforcement guarantee is gone, but the CLI claims it isn't.

**Mitigation needed (currently missing in EXP-001 plan).** Add Step 4-bis: define a schema-validation pass that requires known fields (`VERDICT`, `FINDINGS`, `EVIDENCE-BLOCK CHECK`) to be present AND non-empty. Add Acceptance Criterion: parser fails loudly on stream-json schema drift, never silently passes.

### FM-2: Subagent context-loss makes gatekeeper-g1 verdicts diverge from Claude-Code-native ones in unobvious ways

**Concrete scenario.** When `gatekeeper-g1.md` is invoked via `claude -p --append-system-prompt <vendored-prompt>`, the subagent inherits Claude Code's tools (Read, Bash) but loses the parent conversation's context — including the human's preferences, prior decisions, and project-specific instructions from the running session. It produces verdicts that follow the prescribed format but apply *less* contextual judgment. Examples: it flags an `evidence:` block as "non-generic" because the text is detailed, but a Claude-Code-native subagent would have noticed the text contradicts a decision made earlier in the same session.

**Why this is a real risk.** This is exactly Risk R2 from CTX-002, re-stated specifically. EXP-001 Step 5 + Acceptance Criterion #4 attempt to address this, but the criterion is "exits with appropriate code" and "verdict matches simulation" — both of which are coarse. Subtle judgment-gap divergence won't be caught.

**Why this matters.** chevp-flow's gates would be weaker than Claude Code's. The framework would get *worse* at governance in exchange for becoming portable. That undermines the entire premise.

**Mitigation needed (currently underspecified in EXP-001).** Acceptance Criterion #4 should be sharpened: "verdict from `chevp gate-check G1` against CTX-002 must match the simulation result on at least 80% of *findings*, not just the top-level verdict." Run the comparison on ≥3 different governed plans, not just CTX-002. If divergence is material, treat it as a Kill Criterion trigger.

### FM-3: Bubble Tea + Cobra integration produces "broken-on-ctrl+c" UX failure that destroys first-impression adoption

**Concrete scenario.** A user runs `chevp gate-check G1`, the streaming-claude progress view starts via `tea.Program`, the user hits Ctrl+C mid-stream. The Bubble Tea program receives the interrupt but the Cobra command's parent goroutine doesn't propagate cleanup. Terminal stays in raw mode (`stty raw -echo`); the user has to run `reset` to recover. They post about it on Hacker News. First-impression UX is "broke my terminal".

**Why this is a real risk.** Cobra is synchronous; Bubble Tea wraps an event loop. Cleanup-on-interrupt requires explicit signal handling that is not part of either library's defaults. This pattern bites every Charm+Cobra adopter at least once. EXP-001 Risk row 3 mentions "integration friction" but its mitigation ("use tea.Program only for streaming command") doesn't address the cleanup question.

**Why this matters.** First-impression UX in OSS CLI tooling is brutal. A "broke my terminal" reputation is hard to shake.

**Mitigation needed.** Add explicit Step 10-bis: signal-handler design. `tea.Program.Run` should be wrapped with `defer term.Restore()`; SIGINT must propagate to a context that cancels both the Bubble Tea program and the underlying `claude -p` subprocess.

## Alternative Architectures (engaged, with rejection reasons)

### Alt-A: Pure stdlib — no Cobra, no Charm, just `flag.Parse` + `fmt.Println` + plain ANSI codes for color

**Pros.**
- Zero dependencies → ~3MB binary vs. ~15MB with Charm
- No integration friction (Failure Mode 3 vanishes)
- Trivially debuggable; everything is `print` and `os.Args`
- Fastest cold-start
- Cannot be broken by Charm-ecosystem churn

**Cons.**
- No structured subcommand tree (would need hand-rolled parsing)
- No Huh forms — gate-approval interactivity reduces to `Confirm? [y/N]:` prompts
- No Glamour markdown rendering — verdicts shown as plain text
- Requires reinventing what Cobra/Charm already solve

**Why rejected for EXP-001.** Q5/ADR-003 chose Charm specifically for the Huh-form-to-`AskUserQuestion` mapping (Rule #13). Stdlib-only would preserve correctness but lose the gate-approval ergonomics that motivated the choice. **However**, this remains the reasonable fallback if Failure Mode 3 bites materially during prototyping. EXP-001 should treat "if Charm cost >300 lines for four flows, drop to stdlib" as an active Kill Criterion (it is mentioned, but should be elevated to a checked Acceptance Criterion before proceeding to PRD-001).

### Alt-B: Two-binary split — `chevp-runtime` (stateless, no TUI, pure Pfad-2 wrapper) + `chevp-tui` (orchestrator)

**Pros.**
- Testability: runtime is a pure function with no terminal interaction
- Composability: `chevp-runtime` can be a server backend later (e.g., for a future web UI per PROP-003)
- TUI failure cannot compromise governance (the runtime-only path always works)
- Naturally invites a stable contract between binaries — the Adapter-Contract becomes operationalised in process boundaries

**Cons.**
- Two binaries to ship doubles distribution surface
- IPC overhead (could be sub-process pipe, but adds latency)
- Cobra integration must be solved twice
- Inverts Q7/ADR-005-implication (single module decision)

**Why rejected for EXP-001.** This is fundamentally a v2 decision. Q7 chose single-module for v1. PROP-005 captures the future re-evaluation. Splitting now is over-engineering for a single-adopter dogfooding tool.

**However**, the Challenger notes that this alternative has a property the chosen architecture does not: **the runtime is independently auditable for governance correctness.** Worth revisiting if FM-2 (subagent divergence) bites.

### Alt-C: No prototype — skip EXP-001, write production code directly against the SPEC

**Pros.**
- No throwaway code; everything written becomes v1
- 2-3 weeks faster to v1.0

**Cons.**
- No learning loop. Failure Modes 1, 2, 3 are discovered in production users' terminals.
- Violates Rule #9 (every Exploration produces learning).
- Hypothesis-validation discipline of `exploration-mode: A` is bypassed.

**Why rejected.** The framework's discipline forbids this. The prototype *is* the learning. But it is the strongest argument for keeping EXP-001 lean — the prototype must be small enough that 2-3 weeks is realistic, not a backdoor for de-facto v1 development.

## Counter-Argument (the strongest case against EXP-001)

The strongest counter-argument is this: **the entire chevp-flow project solves a problem the framework's audience does not yet have, with a tool that competes for attention against cleaner alternatives that are improving fast.**

Consider three uncomfortable facts:

1. Claude Code, OpenCode, Cursor, and Aider are all converging on improved process workflows. Cursor's `.cursorrules`, Aider's `--map-tokens`, Claude Code's hooks/skills/subagents — all are claiming the "structured AI-assisted dev" space. chevp-flow could ship a polished v1.0 just as one of these shipped a native lifecycle-management surface, at which point chevp-flow becomes a niche solution to a problem the market has already solved generally.

2. The framework author's stated dogfooding need (CTX-002 Why-now: "the author cannot dogfood the lifecycle outside interactive chat") does not actually require a new binary. He could continue using Claude Code's existing slash commands, which already work. The pain isn't real, it's perceived — "I'd like a CLI" rather than "Claude Code is blocking my work."

3. The "tool-independence proof" is academic until a second adopter materialises. PROP-001 (Pfad-3) and PROP-003 (second adapter) both make clear that's a v2 concern. So EXP-001's actual ROI is: a 2-3 week solo build that demonstrates portability to an audience of one.

**Honest reframing.** EXP-001 exists more for the framework author's intellectual integrity — being able to say "process is tool-independent, and I proved it with a real adapter" — than for measurable end-user value. That's a legitimate motivation. Frameworks-by-construction often start as personal proofs. But it should be **named openly** in the Vision Alignment, not disguised as "validating the framework's portability claim". The Vision Alignment currently reads as if external value is the driver; the Challenger asserts that internal-coherence-for-author is the actual driver, and the plan would be more honest if it said so.

## Product-Coherence Check (engaging Vision Alignment, not rubber-stamping)

EXP-001's Vision Alignment names "Validates the framework's tool-independence claim by producing the first concrete adapter, dogfooding the lifecycle outside Claude Code's interactive surface." Three honest checks against this stated outcome:

### Check 1 — Does chevp-flow advance tool-independence, or merely demonstrate it once?

A single adapter does not prove independence — it proves the framework can be adapted *at least once*. Genuine tool-independence requires either (a) a second adapter that conforms to the same Adapter-Contract (PROP-003) or (b) external adoption signal demonstrating the contract is implementable by someone other than the author.

EXP-001 alone produces neither. The honest framing is **"first step toward tool-independence, not the proof of it"**. Rec: revise EXP-001 §Vision Alignment to say "first concrete instantiation of the Adapter-Contract" rather than "validates the tool-independence claim".

### Check 2 — Does chevp-flow preserve the framework's process discipline, or quietly weaken it?

The chosen Honor-System actor detection (SQ1) plus the Pfad-2 subprocess (which carries Failure Mode 2: subagent context-loss) produces a CLI where gates *technically* run but may *practically* verify less than Claude-Code-native gates.

If chevp-flow ships a "looks-like-a-gate-pass" output that is faster to produce but verifies less rigorously, the framework gets **worse at governance** in exchange for becoming portable. This is the most important coherence risk in the entire project. **It is more important than any technical risk in EXP-001's Risks table**, and EXP-001's Acceptance Criteria do not currently force a side-by-side verdict-fidelity comparison sufficient to detect it.

Rec: elevate FM-2's mitigation (verdict-fidelity comparison on ≥3 governed plans, ≥80% finding match) to a top-tier Acceptance Criterion *and* a Kill Criterion.

### Check 3 — Does the framework actually need its own CLI for the dogfooding claim?

The framework author currently runs the lifecycle via Claude Code; chevp-flow lets him run it via a Go binary. That doesn't change the fact that he is *the only adopter*. Until an external adopter exists, the dogfooding loop is closed within one user's workflow regardless of which tool he uses.

A more impactful Exploration would be: write the Adapter-Contract document (ADR-006 dependency, currently parallel work in chevp-ai-framework), publicise the framework, see if any adopter shows up — *before* building the second tool. If no adopter shows up after 3 months of public visibility, the CLI is solving a problem that doesn't exist; if an adopter shows up, the CLI's design will be informed by their actual needs rather than the author's a priori assumptions.

Rec: consider sequencing — Adapter-Contract document first, public framework launch, then chevp-flow starts when there is at least one external "I would use this" signal. This sequence is honest with the Kill Criterion ("gate-approvals become routine"): if no one outside the author cares, gate-approvals would only have been routine for one person, and the framework was self-pleasing all along.

## Recommendation from Challenger

**G2 should pass only if** Acceptance Criterion #4 (gatekeeper-g1 verdict matches simulation) is sharpened to require ≥80% finding-level match across ≥3 governed plans, AND the Vision Alignment is revised to honestly state "first concrete Adapter-Contract instantiation; tool-independence is claimed but not proven by this work alone".

**Consider deferring chevp-flow** until the Adapter-Contract is public and at least one external adoption signal exists. The current sequencing (build CLI → see if anyone uses framework → maybe build second adapter) is the *least informative* order. The reverse (publish contract → wait for signal → build CLI informed by adopter need) is the *most informative*.

If neither recommendation is adopted, the project will likely succeed *as a personal dogfooding artifact* but should not be marketed as "validating tool-independence" — that claim requires evidence the project does not produce.
