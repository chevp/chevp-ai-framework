Context-Files as a Wiki

Flat IDs and Folder-per-Plan in a Docusaurus Subsite

chevp · 2026-05-02 · 6 min read · Companion lab
Abstract

Plans, ADRs, and specs in chevp-ai-framework are currently kept as loose markdown files in context/, identified by §-numbers that change as the document moves between Context, Exploration, and Production. This paper argues that the §-scheme conflates identity with position, and proposes a small structural change: a flat ID per plan (FW-01, ADR-007, PROP-014), a folder per plan containing one markdown per phase, and a Docusaurus subsite served at /flow/ next to the existing static site. The result is one stable URL per artefact across its lifetime, real navigation, and zero risk to the existing pages.

1.Introduction

The framework's atomic unit is the plan. A plan is born in Context, matures in Exploration, and is implemented in Production — three phases that today live as three separate files with three different §-numbers. Renaming on every transition is friction, and worse, it breaks the only property an identifier should have: persistence.

This paper proposes one change. The plan keeps a single ID for life. The phases become files inside a folder named after the ID. Everything else — sidebar, search, citable URLs — falls out of treating the result as a small wiki rather than a flat directory.

2.The §-Scheme and What Breaks

Existing plan IDs encode where the plan sits in the document tree (§3.2.1). When a plan moves between phases, its §-number moves too. Three things follow:

  • cross-references between artefacts go stale on every transition;
  • the same plan has no canonical URL — it is three URLs, none of which is "the" plan;
  • retrospective tools (audits, diffs, changelogs) cannot trace a plan's history without external indexing.

Identity has been smuggled into position. The fix is to separate them.

3.Proposal: Flat IDs, Folder-per-Plan

Each plan is assigned one flat, monotonically issued ID at creation time (FW-01, FW-02, …). The ID never changes. A plan owns a folder. Inside, one file per phase (context.md, exploration.md, insights.md, production.md). Phases that have not happened yet simply don't exist as files.

4.The Wiki Layer

Files alone do not make a wiki. The proposal adds a thin Docusaurus subsite that consumes the folder structure, generates a sidebar grouped by lifecycle status, and produces stable URLs. Critically, the subsite is built into docs/flow/ — the existing static pages in docs/ are untouched.

Tab. 1 — Both sites coexist on GitHub Pages under different paths.

Identity is the folder. Position is the sidebar. Phase is the filename.

5.Trade-offs

Why do it

  • One stable URL per plan, citable for life.
  • The lifecycle becomes navigable, not just nameable.
  • Search and sidebar come for free.
  • Existing pages cannot break — the subsite is additive.

What it costs

  • A second build pipeline (Docusaurus) to maintain.
  • An ID-allocation step every time a plan is created.
  • Migration of existing §-numbered plans is non-trivial.
  • Two URL schemes coexisting means two places to look.

6.Kill Criteria

The Lab is the testbed. The proposal is rejected if, after running it for one full plan end-to-end (Context → Exploration → Production), any of the following hold: cross-links rot anyway; the build pipeline costs more attention than the §-renames it replaces; or the Docusaurus layer makes plans harder to write than plain markdown. The §-scheme stays intact until a real plan has lived inside the new structure and survived.

7.Related Work

ADR-001 of the framework establishes content-oriented governance. The Lab is the executable companion to this paper — it is where the proposal stops being a paper and starts running. Diátaxis (Procida) and arc42 argue from the same root: documentation is shaped by what it is for, not by where it sits.

8.Conclusion

Renaming a plan on every phase transition is a small annoyance with a large bill: the plan loses its identity. A flat ID, a folder per plan, and a thin wiki layer pay that bill once and give back stable URLs, real navigation, and a place for the lifecycle to live. The Lab is already running this experiment. This paper is the abstract.
